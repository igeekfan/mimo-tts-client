## Commands

```bash
# Desktop (Wails) — Go backend + embedded frontend, hot reload
wails dev
wails build                                   # → build/bin/mimo-tts-client

# Web mode — same core, served as HTTP API + SPA (build tag `web`)
go build -tags web -o tts-server .            # needs frontend/dist to exist (npm run build)
./tts-server                                  # listens on :8080 (override TTS_WEB_ADDR)
docker build -t tts . && docker run -p 8080:8080 -e TTS_API_KEY=key tts

# Go
go build ./...                                # NOTE: default build tag = desktop (main.go), not web
go vet ./...
go test ./...                                 # unit tests live in internal/core/tts_test.go
go test ./internal/core -run TestSynthesizeSpeech   # single test

# Frontend (from frontend/)
npm install
npm run build                                 # tsc type-check THEN vite build
npm run dev                                   # Vite only (no Go backend)
```

Always run `go build ./...` and (from `frontend/`) `npm run build` before committing.

## The one architecture concept that matters: dual mode

The app compiles into two products from one codebase, selected by Go **build tags**:

- **Desktop** (`//go:build !web`, `main.go`) — Wails wraps `desktop.App`, embeds `frontend/dist`,
  and exposes Go methods to JS via generated `frontend/wailsjs/` bindings.
- **Web** (`//go:build web`, `main_web.go`) — a plain `net/http` server (`internal/httpapi`)
  exposes the same operations as `/api/*` REST + SSE, and serves `frontend/dist` as an SPA.

Both paths are thin adapters over the **single source of business logic in `internal/core`**
(`Service`). Neither `desktop/` nor `internal/httpapi/` should contain TTS/DB logic — if you find
yourself adding it there, it belongs in `core`. `core.Service` talks to the outside world only
through `core.Hooks` (e.g. `AppLog`), which each mode wires up differently (Wails events vs SSE hub).

### The frontend mirrors this split in ONE place: `frontend/src/lib/backend.ts`

Every backend call in the app goes through `backend.ts`, which detects the mode at **runtime** by
checking for `window.go.desktop.App`. Each exported function branches:
desktop → Wails bind call, web → `fetch('/api/...')`. **App code and components import only from
`../lib/backend` (and `../lib/runtime` for events) — never from `wailsjs/` directly.** This is the
rule that keeps web mode working. `frontend/wailsjs/` is generated by Wails; do not edit it.

### Streaming crosses this boundary too

Real-time PCM16 streaming is exposed to the UI as one async generator
(`SynthesizeSpeechStream` in `backend.ts`), but the transport differs by mode:
- Desktop: `StartSynthesizeSpeechStream` returns immediately; chunks arrive as Wails events named
  `tts:stream:<streamId>` and are re-assembled into the generator.
- Web: an SSE response body (`data: <base64>` frames, terminated by `[DONE]`).

If you add a streaming feature, you must implement **both** transports or web/desktop will diverge.

## MiMo TTS API notes (`internal/core/tts.go`)

The upstream API (`https://api.xiaomimimo.com/v1/chat/completions`) is OpenAI chat-completions-shaped,
auth via the `api-key` header (key from settings, falling back to `TTS_API_KEY` env). The request
message shape is **model-dependent** — see `buildMessages`:
- Preset / clone models: `user` = style tag (may be empty), `assistant` = the text to speak, and
  `voice` goes in the `audio` config.
- `mimo-v2.5-tts-voicedesign`: `user` = the voice *description*; the spoken text is the `assistant`
  message, and there is no `audio.voice`.

Non-streaming returns base64 WAV; streaming returns raw PCM16 (24kHz mono) — `addWavHeader` wraps
PCM into a playable WAV when the response isn't already `RIFF`.

## Data & config

- SQLite via GORM, stored under `os.UserConfigDir()/mimo-tts-client` (`GetDataDir`). Holds settings
  and synthesis history (audio blobs included).
- Settings auto-save on every change; API key and base URL are configurable in the UI or via env.
- All user-facing strings are i18n'd (zh-CN / en-US) on both backend (`core/i18n.go`) and frontend
  (`useI18n()` hook). Add new strings to both locale files.
